# Overview

Dynamic AI is a processing system that reads GTR2 race results, extracts timing data, fits hyperbolic curves (T = a/R + b) to AI performance, and updates AIW files to adjust AI difficulty. 

The goals for this program were:
- Different AI levels on quali and race (In my experience qualy was much harder than the race, so a good qualy always meant a boring race)
- Getting values that are not perfect (GTR2 has other variables that influence AI laptime) but make for challenging qualy sessions and races with as much fight as possible.

The architecture follows a modular design with clear separation: data extraction, persistence, mathematical modeling, AIW manipulation, and GUI.

Also this Architecture description has been generated by AI but reviewed by AA.

## Core Data Flow

- File Monitor Daemon (gui_file_monitor.py) polls raceresults.txt at configurable intervals (default 5s). On file change, it reads the file and passes raw data to DataExtractor.

- Data Extraction (core_data_extraction.py) parses [Race] section, finds Scene= (track path) and AIDB= (AIW path). It then scans each [SlotN] for driver info, qualifying time, best lap. User is slot 0; AI are slots 1+. Lap times are converted from MM:SS.mmm to seconds. The extractor also reads the referenced AIW file to get current QualRatio and RaceRatio.

- Database Layer (core_database.py) stores:
  - data_points: (track, vehicle_class, ratio, lap_time, session_type) - each point is one AI driver's time at a specific ratio
  - race_sessions: metadata about each race event
  - ai_results: per-AI driver results for each race
  - formulas: fitted curve parameters per (track, vehicle_class, session_type)
  - user_laptimes: historical user lap times per combo, automatically trimmed to nr_last_user_laptimes
    
- Vehicle Classification (core_vehicle_scanner.py, vehicle_classes.json) maps vehicle names to classes. This allows grouping different cars (e.g., "Ferrari 458", "Audi R8 LMS") under the same class so formulas apply across similar vehicles. The scanner reads .car files from GameData/Teams/ and extracts Description= fields.

- Mathematics (core_math.py) is the single source of truth for all mathematical operations. It provides:
  - time_from_ratio() and ratio_from_time() for hyperbolic calculations
  - clamp_ratio() and clamp_b() for value validation
  - fit_hyperbolic() for curve fitting using scipy.optimize.curve_fit
  - Outlier detection methods (std, IQR, percentile)
  - All other modules import math functions from here to ensure consistency

- Autopilot Engine (core_autopilot.py) is the core logic. It maintains FormulaManager for persistent formula storage and AutopilotEngine for calculations. When new race data arrives:
  - Gets existing formula for (track, class, session) or creates one with default a=32.0
  - Retrieves median user lap time from history (if available) to smooth outliers
  - Calculates new ratio using ratio_from_time() from core_math
  - Applies min/max clamping (cfg.yml: min_ratio, max_ratio)
  - Updates AIW file via update_aiw_ratio()
  - Saves the point to database and updates formula using fit_hyperbolic() from core_math

- AIW Utilities (core_aiw_utils.py) handle case-insensitive AIW file location (since GTR2 paths can be case-inconsistent). It searches by path variants, case-insensitive globbing, and stem matching. When updating, it first ensures both QualRatio and RaceRatio exist in the file (adding default 1.0 if missing), then uses regex replacement to update the value. Backups are created in aiw_backups/ before first modification.

- User Laptimes (core_user_laptimes.py) stores each user lap with the ratio active at that time. It supports median calculation to provide a stable target, reducing the impact of single outlier laps.

## Startup Sequence (dyn_ai.py)

- Initialize rotating log handler (core_log_manager.py - keeps last 2 log files)
- Load config from cfg.yml with defaults (core_config.py)
- Create database if missing
- Run pre-run checks (gui_pre_run_check.py):
  - Validate vehicle_classes.json exists and has proper structure
  - Validate GTR2 base path
  - Check GTR2.exe exists
  - Check PLR file has Extra Stats = "0" (required for correct timing)
  - Verify vehicle cache exists (created by setup manager)
- Launch MainWindowTk (Tkinter main window)

## GUI Components

- Main Window (gui_main_window.py): Tkinter-based dashboard showing current ratios, AI ranges, user times. Manages file monitor daemon and triggers autopilot processing. Now modularized into:
  - gui_track_selector.py: Track selection dialog for manual track picking
  - gui_ratio_panel.py: Reusable ratio display panels for qualifying and race sessions
- Visualizer (dyn_ai_visualizer.py, gui_curve_graph.py, gui_session_panel.py): PyQt5-based graph viewer showing hyperbolic curves, data points, and user history. Supports manual formula editing and auto-fitting.
- Setup Manager (dyn_ai_setup.py): Tkinter-based configuration, vehicle class management, data point editing, backup restore, and log viewer.

NOTE: we use tkinter everywhere but on visualizer, where we use PyQt5. PyQt5 is too heavy but tkinter cannot do the graph work in the same way.

## Configuration (cfg.yml)

base_path: /path/to/GTR2 - The path to your actual GTR2 installation
db_path: ai_data.db - The Database that saves all the data this program uses.
max_ratio: 1.5 - Maximum Ratio to accept. GTR2 AI behaves weird above a given value, and you can adapt it at will. Default 1.5 should be ok.
min_ratio: 0.5 - Same as above but for the lower level
nr_last_user_laptimes: 5 - How many of your Laptimes to consider when calculating the Ratio? 1 would be "just the latest one"
outlier_method: std - Method to find out a laptime is too weird to use, Leave as is for now
outlier_min_points: 3
outlier_threshold: 2.0
poll_interval: 5.0 - Program looks for raceresults every <poll_interval> seconds.

## Mathematical Model

The core relationship: LapTime = a / Ratio + b

- a (fixed at 32.0 by default, user-adjustable) controls curve steepness
- b (asymptote) represents theoretical minimum lap time as Ratio → ∞
- The model assumes linear relationship between 1/R and lap time
- Fit quality is tracked via confidence, avg_error, max_error
